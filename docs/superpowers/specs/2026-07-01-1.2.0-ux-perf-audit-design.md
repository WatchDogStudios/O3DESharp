# O3DESharp 1.2.0 — UX/UI + Performance Audit & Release

**Date:** 2026-07-01
**Branch:** `development`
**Status:** Approved by maintainer (mikaelamanfo@wdstudios.tech) — scope: all 17 items below.

## Goal

Audit O3DESharp for UX/UI and performance improvements, fix what's found, ship as
**v1.2.0** (current released version is v1.0.0; there was no v1.1.0). The maintainer
is separately refactoring the BehaviorContext reflection/dispatch control surface for
**v1.3** — any finding that touches that control surface's core design is explicitly
out of scope here and is fed forward to that refactor instead.

Audit method: a 6-area multi-agent sweep (runtime C++ perf, editor/inspector UX, C#
API ergonomics, build/onboarding blockers, binding generator, docs), each finding
independently re-verified against current code (not assumed from prior audits — 30/32
claims survived verification). Full raw findings live in the workflow transcript; this
doc captures the ship-scoped subset with enough detail to plan and implement from.

## Non-goals

- No changes to `GenericDispatcher`'s BehaviorContext lookup/dispatch internals, the
  marshaling table's core design, or `NativeReflection.InvokeStaticMethod` /
  `InvokeInstanceMethod` / `InvokeGlobalMethod` / `GetProperty` / `SetProperty` /
  `BroadcastEBusEvent` / `SendEBusEvent` internals — reserved for the 1.3 refactor.
- No `CoralHostManager` AssemblyLoadContext restructuring (hot-reload aliasing) — too
  architecturally invasive for a point release; tracked separately, post-1.2.0.
- No libclang PCH/prelude-caching work for the binding generator's clang backend —
  large effort, and that backend is no longer the default path (reflection is).
- No FirstPersonController work — that's separate, currently-paused work on
  `feat/fpc-port-m1`, untouched by this effort.

## Scope: 17 items to ship in 1.2.0

Grouped by workstream so they can be implemented largely independently.

### A. C# math/API correctness & ergonomics (`Assets/Scripts/O3DE.Core/`)

1. **`Quaternion.LookRotation` returns the conjugate (inverted) rotation** —
   `Math/Quaternion.cs:269`, Shepperd conversion at `:286-330`. **Critical / ship-blocking.**
   Verified: produces backwards-facing rotations for all non-trivial directions (turrets,
   cameras, characters facing a target). Only the trivial forward=Up case is unaffected,
   which is why it passed prior manual checks. Fix: correct the Shepperd trace/branch
   formulas to consume the transpose of the (right, forward, up) basis so it's consistent
   with `RotatePoint`'s convention (same convention native `Transform.Forward/Right/Up`
   uses). Add a unit test asserting `LookRotation(dir).RotatePoint(Vector3.Forward) ==
   dir.Normalized` for several non-axis-aligned directions — there are currently zero
   tests covering this function.
2. **`Vector3`/`Quaternion`/`Vector2` `Equals`/`GetHashCode` contract violation** —
   `Math/Vector3.cs:346`, `Quaternion.cs:482-499`, `Vector2.cs:195-210`. `Equals` uses
   epsilon tolerance, `GetHashCode` uses exact float bits — breaks `Dictionary`/`HashSet`
   lookups for near-equal keys. Decision: drop epsilon from `Equals` (exact equality,
   matches `System.Numerics.Vector3` convention) rather than quantizing the hash, since
   it's simpler and matches common expectation. Document the divergence from Unity's
   exact-equality-only convention in an XML doc comment.
3. **`Debug.Log`/`LogWarning`/`LogError`/`LogDebug` crash on a bad format string** —
   `Debug.cs:47,75,103,194`. Wrap `string.Format` in try/catch in all four overloads;
   fall back to the raw format string plus a `[format error]` note instead of throwing
   `FormatException` uncaught.
4. **`Debugger.WaitForAttach(TimeSpan.Zero)` means "wait forever"** — `Debugger.cs:72`,
   logic at `:77`. Inverts standard .NET zero-timeout semantics and contradicts the
   method's own XML doc ("`TimeSpan.Zero` means check once") — corroborated by
   `SCRIPTING_GUIDE.md`'s own troubleshooting FAQ ("Editor freezes after WaitForAttach
   and never resumes"). Fix: `TimeSpan.Zero` means check once; only
   `Timeout.InfiniteTimeSpan`/negative spans mean wait forever.
5. **`NativeReflection.SerializeArgumentToObject` silently stringifies unrecognized
   argument types** — `Reflection/NativeReflection.cs:572`. Concrete bug today: `Vector2`
   isn't in the explicit switch, so it serializes as a display string instead of the
   numeric array the native marshaler expects, producing a confusing type-mismatch
   several layers away from the real cause. This is a narrow, C#-side-only fix adjacent
   to (not inside) the dispatch internals: change the `_ => arg.ToString()` fallback to
   `throw new NotSupportedException($"Unsupported argument type {arg.GetType()} in
   {callingApi}")`.

### B. Runtime C++ perf (`Code/Source/Scripting/`)

6. **Unconditional `AZStd::string::format` on the hot dispatch path** —
   `GenericDispatcher.cpp:2046` (+2089, 2262, 2288, 2374, 2453). 6 of 7 cited call sites
   build an error-path diagnostic label unconditionally, even on the success path, on
   *every* C# reflection call. `DispatchEBusEvent` already does this correctly with a
   lazy `makeError` lambda — mirror that pattern at the other 6 sites.
7. **`Entity.GetChildren()` is O(n²) from script** — `ScriptBindings.cpp:249,257`
   (`Entity_GetChildAtIndex`/`Entity_GetChildCount`) and the C# wrapper
   `Entity.cs:161-179`. Each call re-fetches the full children vector via EBus; the
   wrapper calls `GetChildCount` once then `GetChildAtIndex` in a loop. Add one bulk
   internal call `Entity_GetChildren` returning a marshaled array in a single call;
   update `Entity.cs` to use it instead of the loop.
8. **`StackAllocator` does per-object heap `new`** —
   `Marshaling/BehaviorContextMarshaling.cpp:209`. Contradicts its own name/doc ("flat
   arena"); every JSON-marshalled string argument does a `new AZStd::string()` plus
   vector bookkeeping. Swap to a real bump/arena allocator (fixed inline buffer +
   fallback) for one marshaling pass's string/buffer storage.
9. **`ForwardEventToManaged` re-resolves the Coral type by name every event fire** —
   `GenericDispatcher.cpp:1887`. The resolved `Coral::Type*` never changes at runtime;
   cache it once (at `RegisterEBusHandler` time, or lazily on first call) instead of a
   string-keyed hashmap lookup per event.

### C. Editor / inspector UX (`Code/Source/Tools/`, `Editor/Scripts/`)

10. **C# Project Manager Build/Build All freezes the editor UI, no timeout** —
    `Editor/Scripts/csharp_project_manager.py:1438`. `build_project()` calls
    `subprocess.run` synchronously on the Qt UI thread from three call sites
    (`_build_project`, `_build_all`, `ScriptBrowserDialog._build_selected_project`), no
    timeout — a hung `dotnet build` freezes the editor indefinitely with no way to
    cancel. The fix pattern already exists and is proven in the same file for the
    auto-build-after-generation flow: `_BindingBuildWorker(QThread)` with streaming log
    + `proc.wait(timeout=300)`. Route all three UI build actions through it. **Highest-
    value editor UX fix in this release.**
11. **"Clear Selection" script picker no-ops + prints a misleading warning** —
    `Code/Source/Tools/Components/EditorCSharpScriptComponent.cpp:269` and
    `CSharpScriptClassPropertyHandler.cpp:160-176`. Cancel and explicit-clear are
    conflated as the same empty-string return across the Python/C++ bus boundary.
    Fix: in `_select_none()` distinguish explicit-clear (return `""`) from cancel
    (return `None`); update `OpenScriptPicker`/`OnBrowseScript`/
    `CSharpScriptClassPropertyHandler` to treat them differently — clear blanks the
    field and re-validates, cancel is a true no-op with no warning.
12. **`ValidateScriptClass` — richer validator implemented but never wired up** —
    `EditorCSharpScriptComponent.cpp:199`. The Python-side validator (distinguishes
    bad file path / wrong base class / "found but not compiled") exists and works but
    is orphaned; `ValidateScript()` reimplements a strictly worse native-only check.
    Wire `ValidateScript()` to call `CSharpEditorToolsBus::Events::ValidateScriptClass`
    with fallback to the existing Coral-only check.
13. **Stale "dormant" comment on `CSharpExposedProperties` handler** —
    `O3DESharpEditorSystemComponent.cpp:201` and `.h:149-154`. The feature has been
    live since Phase 14; the comment still says it's inert Phase-10 scaffolding.
    Delete/update the comment block.

### D. Onboarding / build (`Code/Tools/BindingGenerator/`)

14. **BindingGenerator targets net8.0, README only documents net9.0 SDK** —
    `O3DESharp.BindingGenerator/O3DESharp.BindingGenerator.csproj:5`. First-run
    onboarding blocker: `O3DESHARP_AUTO_GENERATE_BINDINGS` (default ON) drives a CMake
    target that runs `dotnet build`/`dotnet run` against this net8.0 project; a new dev
    who installs only .NET 9.0 SDK per the README hits an undocumented framework-
    resolution failure. Fix: retarget `O3DESharp.BindingGenerator.csproj` and
    `BindingGenerator.Tests.csproj` to net9.0 (brings the whole managed toolchain to one
    TFM at the source, rather than just documenting the gap). Watch for any net8-only
    API usage during the bump (unlikely — ClangSharp/System.Text.Json are compatible).

### E. Binding generator quality-of-life (`Code/Tools/BindingGenerator/`)

15. **`binding_config.json` parse failures silently fall back to defaults, exit 0** —
    `Configuration/BindingConfigLoader.cs:46`, misleading console line at
    `Program.cs:410` (still prints "Configuration: {path}" as if it loaded). A JSON
    typo silently discards all custom gem settings and proceeds against defaults
    successfully-looking. Fix: keep the fallback-to-defaults behavior (don't change the
    exit code — avoid breaking existing automated invocations in a point release), but
    make the diagnostic unambiguous: replace the current success-looking "Configuration:
    {path}" line with a clear `WARNING: failed to parse {path} ({exception message}) —
    using default configuration` when the load fails.
16. **Binding generator diagnosability + determinism bundle** (grouped, same
    workstream):
    - Zero-bindings result gives no actionable reason at default verbosity —
      `MultiGemBindingGenerator.cs:195`, skip logs at `O3DEHeaderParser.cs:547,658`.
      Track aggregate skip-reason counters, print an unconditional one-line summary.
    - No schema validation for `binding_config.json` — `BindingConfigLoader.cs:49`.
      After deserializing, diff raw JSON keys against `BindingConfig`'s known
      properties (case-insensitive), warn on unrecognized keys and unresolved
      `${VAR}` references.
    - Non-deterministic output ordering — `MultiGemBindingGenerator.cs:324,400`.
      `Directory.GetFiles` order is filesystem-dependent and nothing downstream sorts;
      causes cosmetic diffs across machines and a non-deterministic tiebreak in
      `DeduplicateByName` on a name collision. Sort header file list
      (OrdinalIgnoreCase, full path) and sort `bindings.Classes/Functions/Enums` by
      qualified name after parsing.
    - If time allows: share a single `CXIndex` across all `ParseHeaderFile` calls for
      a gem in `O3DEHeaderParser.cs:250,261` (the small, low-risk half of the
      clang-backend perf issue — full PCH work is out of scope, see Non-goals).

### F. Docs accuracy

17. **Docs describe ClangSharp as canonical when the actual default is the reflection
    backend** — `README.md:556`, `GENERATED_BINDINGS_GUIDE.md:69`,
    `SCRIPTING_GUIDE.md:1002`. Three independent layers (CLI default, CMake target,
    Python config) agree reflection (`--source reflection`, reading
    `reflection_data.json`) is the real default, but every doc's quick-start commands
    omit `--source` entirely. One coordinated doc pass: add a "Which backend should I
    use?" callout, update example commands to explicitly show `--source clang` or
    `--source reflection` (with its prerequisite — launch the editor once first —
    documented). Bundle in while touching these docs:
    - `README.md:650` references a non-existent "Tools > C# Script Manager" menu —
      correct to the real dialog title "C# Project Manager" and note how it's actually
      opened today (menu registration is currently a stub).
    - `SCRIPTING_GUIDE.md:706,1149` and `README.md:622-627` omit `Guid`/`AZ::Uuid`
      EBus-handler marshaling support that already shipped
      (`EBusHandlerRegistry.cs:294-298`) — add it to the marshaled-types list.
    - `GENERATED_BINDINGS_GUIDE.md:51` Prerequisites implies the generator itself is
      net9.0 — resolves itself once item 14's csproj retarget lands.

## Release mechanics

1. Implement all 17 items on `development` (small, mostly-independent commits grouped
   by workstream — one commit per numbered item or tightly-related pair, not one giant
   commit).
2. Verify: build the gem, run existing test suites (BindingGenerator xUnit tests, any
   C# unit tests for `O3DE.Core` math types — add the `LookRotation` regression test
   here per item 1 — Python editor tests, SmokeTests console app if relevant).
3. Bump `gem.json` `"version"` from `1.0.0` to `1.2.0`.
4. Write a `RELEASE_NOTES.md` v1.2.0 section following the existing v1.0.0 section's
   structure (Highlights / Bug fixes / Documentation / Tooling-CI / Known limitations /
   Upgrade notes), commit message style matching repo convention (no Claude/Anthropic
   attribution trailer — see commit_conventions memory).
5. Confirm with the maintainer before: tagging `v1.2.0`, merging `development` → `main`,
   and pushing to remote. These are the hard-to-reverse/shared-state steps and need an
   explicit go-ahead at that point even though the maintainer has pre-approved "full
   release" as the end goal.

## Testing strategy

- **Item 1 (LookRotation):** new unit test is mandatory before merge — this is the
  ship-blocking bug and currently has zero coverage.
- **Items 2-5 (C# API):** add/extend unit tests where `O3DE.Core` has an existing test
  project; otherwise a manual verification note is acceptable given the smallness of
  each fix.
- **Items 6-9 (C++ perf):** these are behavior-preserving optimizations — existing
  tests (if any cover this path) must still pass; no new test infra required unless
  trivial to add.
- **Items 10-13 (Editor/Python UX):** manual verification in the O3DE editor is the
  primary check (Python editor test suite covers what it covers; these are UI/dialog
  behaviors that are hard to unit test).
- **Item 14 (TFM retarget):** verify `dotnet build` succeeds against a machine with
  *only* .NET 9.0 SDK installed, matching the README's documented prerequisite.
- **Items 15-16 (Binding generator):** BindingGenerator's existing xUnit suite should
  cover config-loading and determinism; add cases where missing.
- **Item 17 (Docs):** review-only, no automated check.

## Open questions / risks

- Item 2's `Equals`/`GetHashCode` fix (exact equality) is a very minor behavior change
  for any existing script code that relied on epsilon-tolerant `Equals` semantics —
  acceptable since it fixes a genuine contract violation, but worth calling out in
  release notes' "Upgrade notes" in case anyone was implicitly depending on it.
- Item 1's fix touches a widely-used math primitive; the plan calls for verifying
  against `RotatePoint`'s convention and adding tests, but this is the single item
  most worth a careful second look before merge given its blast radius.
